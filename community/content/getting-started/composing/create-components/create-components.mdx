# Create Components

When you create a component you generate its initial source files and add its ID to the `.bitmap` file.

Components can be of various types such as, React, Angular, NodeJS, and others. The type of each component is determined by the Env it uses.

Bit's official Envs offer Component templates to help you create new components. Run the following inside your Workspace to list all available component templates:

```bash
bit templates
```

The output is similar to the following (it may also include templates provided by custom Envs in your Workspace):

```
teambit.react/react
    react (a basic react component)
    react-context (a react context component)
    react-hook (a react hook component)
    react-env (customize the base React env with your configs and tools)

teambit.harmony/node
    node-env (customize the base Node env with your configs and tools)
    module (a general purpose node module)

teambit.harmony/aspect
    aspect (extend Bit capabilities)

teambit.mdx/mdx
    mdx (Content component)
```

Run the following to create a component using one of these templates:

```bash
bit create COMPONENT_TEMPLATE COMPONENT_NAME [--scope SCOPE_NAME]
```

For example, run the following to create a React component:

```bash
bit create react ui/my-welcome
```

The output is the following:

```bash
1 component(s) were created

company.demo/ui/my-welcome
    location: base-ui/ui/my-welcome
    env:      teambit.react/react
```

:::note
By default, components are created according to the `defaultScope` defined in the [workspace.json](/workspace/workspace-json). 
Specifying a different scope can be done with the `--scope` flag while using `bit create`.
:::

Components are set to their development environment, according to their namespace as defined in `workspace.json`.
Default [workspace configuration](/workspace/workspace-json) usually include [Variant rules](/) namespaces such as `ui`, `envs` or `pages` to be associated with their corresponding envs.

```json title="workspace.json"
  "teambit.workspace/variants": {
    // teambit.harmony/node is the default environment. component under different namespaces than the below will fall back to it.
    "{ui/**}": {
      "teambit.react/react": {}
    },
    "{content/**}": {
      "teambit.react/react": {}
    },
    "{envs/**}, {extensions/**}": {
      "teambit.harmony/aspect": {}
    }
  }
```

Let's create our first React component. In this example, components created with the `ui` namespace are automatically set to the React env.

```bash
bit create react ui/my-welcome
```
```
1 component(s) were created

company.demo/ui/my-welcome
    location: base-ui/ui/my-welcome
    env:      teambit.react/react
```

That's it! Bit has generated the component in a new [Component Directory](/workspace/component-directory) in the Workspace. You should think of every component created in Bit as an independent module which can be used from anywhere.

## Component directories
Let's view the newly created component directory.

```bash
└── ui/my-welcome # Component directory
    ├── index.ts # Main component file.
    ├── my-welcome.tsx # Implementation of the component.
    ├── my-welcome.spec.tsx # Test file of the component.
    ├── my-welcome.composition.tsx # Compositions for simulating the component in different states.
    └── my-welcome.docs.mdx # Docs file for the component.
```

As seen above, the created component has few different files generated by default.

- `index.ts` / [Component main file](/components/main-file), exposing the component API.
- `my-welcome.tsx` / Implantation of the React component. You can add as many files as required for your Component to operate.
- `my-welcome.spec.tsx` / [Tests](/tester/overview) file for the Component.
- `my-welcome.compositions.tsx` / [Compositions](/compositions/overview) file helping to simulate and visualize the component in different states.
- `my-welcome.docs.md` / [Docs](/docs/overview) file. A markdown file for documenting the component.

You can choose to not include tests files, docs files or even the composition files, however we highly recommend in including all of these as it encourages best practices and makes it easier for other developers to understand and trust your component.

## Inspecting and browsing components

You can inspect the component to see how Bit how it is configured, obtain access to the component metadata and its configuration using the `bit show` command.
Per the example above, we can now verify that the example is associated to the desired React env.

```bash
bit show ui/my-welcome
```
